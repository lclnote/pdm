# PDM (Project Development Manager) 要件・基本設計書と現状の実装状況の対比・課題洗い出しレポート

本レポートは、要件定義書（`PDM.md`）および基本設計書（`docs/basic-design.md`）に記載されているシステム要件を、現在の実装状況と照らし合わせて「完了している機能」「未実装/要件との相違箇所（課題）」として網羅的に洗い出し、整理したものです。

---

## 1. プロジェクト管理
### 【実装済・完了】
- **プロジェクトCRUD**: 新規作成、編集、削除、一覧表示機能。
- **プロジェクト期間設定**: 開始日・終了日の設定と、カレンダー日付ピッカーの提供。
- **進捗率計算ルール（2パターン対応）**:
  - `task_count`（タスク数基準）: `完了リーフタスク数 / 全リーフタスク数` を重み（`weight`）を加重して計算。
  - `hour`（工数基準）: `完了タスクの実績工数合計 / 全タスクの見積工数合計` で計算。
  - 親タスクの進捗率は末端リーフ（子を持たない孫や末端タスク）のみから算出し、中間タスクを排除。
  - プロジェクト・フェーズ・タスクの全階層で上記のルールに従って一貫して計算され、ダッシュボードに反映。
- **期間整合性警告（警告の伝播）**: プロジェクト期間外のフェーズ、フェーズ期間外のタスクを検知し、ダッシュボードおよび詳細に警告をアイコン（⚠️）とツールチップで表示。

### 【未実装・相違箇所】
- 特になし。要件をすべて100%充足して実装済み。

## 2. フェーズ（Phase）管理
### 【実装済・完了】
- **フェーズCRUD**: 追加、編集、削除、一覧表示機能。
- **並列フェーズ対応**: フェーズごとに並行実行フラグ（`parallel_execution`）を切り替え可能。
- **フェーズ順序制御 (Drag & Drop)**: `PhasesPage` で直感的にドラッグ＆ドロップして並び替え、データベースの `sort_order` が一括で永続更新される機能を搭載。
- **ゲート管理（簡易申請・承認）**: 
  - フェーズが `active` の時に「完了申請」理由・エビデンスを送信してステータスを `gate_waiting`（ゲート通過待ち）へ変更可能。

### 【未実装・相違箇所（課題）】
- **ゲート申請と `applications` テーブルの不一致**:
  - 現状のフェーズの「ゲート申請」は `POST /api/phases/{id}/gate-request` を呼び出して `phase.status = "gate_waiting"` とする簡易ロジックで、`applications`（申請履歴テーブル）にはレコードが作成されません。
  - PMによる承認・却下は、フェーズ編集モーダルからステータスを手動で `completed` や `active` に戻す形で実現されています。「申請管理画面」にはフェーズゲート申請レコードは表示されません。

## 3. タスク管理（3階層）
### 【実装済・完了】
- **3階層タスクツリーのレンダリング**: 親子・孫階層をインデントおよび再帰ツリーでTasksPageに描画。
- **タスク詳細パネル**: 選択したタスクの詳細、担当者、見積/実績工数、状態、警告の表示。
- **重み・実績工数管理**: 進捗率に加重する `weight` および実績工数 `actual_hours` の表示・設定・編集機能。
- **担当者・協力者（複数）の管理**: タスクへの担当者1名、および協力者（タスクコラボレータ）複数名を追加・削除できるUIとAPI。
- **担当者・協力者選択の所属メンバー制限**: 担当者や協力者の検索・補完リストを、プロジェクトに属するメンバーのみに制限するセキュリティフィルタを搭載。
- **タスク完了申請**: ステータスが対応中のタスクに対し、完了理由・成果物エビデンスを入力して「完了申請」を送信し、状態を `awaiting_approval` に移行させて `applications` レコードを作成。

### 【未実装・相違箇所（課題）】
- **保留申請・中断申請のUI未搭載**:
  - `PDM.md` では担当者が「保留申請（hold）」や「中断申請（suspend）」を行い、PM承認を経て状態が `on_hold` や `suspended` に移行するフローが定義されていますが、フロントエンドのタスク詳細には「完了申請」ボタンのみが実装されており、保留・中断申請の送信UIはありません。
- **タスク階層ごとの担当者ロール制限の未強制**:
  - 「子を持つタスク担当者はサブリーダー」「リーフ担当者は作業者」とするロール定義が記載されていますが、DBやフロントエンド側でロールに応じた担当者アサイン制限は強制されておらず、すべてのプロジェクトメンバーをアサイン可能です（運用上カバーする状態）。

## 4. スケジュール管理 & 依存関係管理
### 【実装済・完了】
- **期間設定**: 各フェーズ・タスクレベルでの開始日・終了日（期限日）の設定・変更機能。
- **期間外警告（警告Cascade）**: 親の期間をはみ出たスケジュールを警告する整合性チェック機能。
- **依存関係データベース構造**: 先行・後続関係を表す `task_dependencies` テーブル。

### 【未実装・相違箇所（課題）】
- **ガントチャートUI（Level 5）**:
  - `docs/basic-design.md` で言及されている `dhtmlx-gantt` などのガントチャートは未導入。現状の TasksPage は階層ツリーグリッドでの一覧表示のみです。
- **依存関係設定UI & 矢線表示 & クリティカルパス算出（Level 5）**:
  - 依存関係のループ検出、クリティカルパスの自動算出、およびガントチャート上での依存関係矢線の表示は未実装です。
- **カレンダー連携（Level 5）**:
  - 祝休日設定や営業日ベースの期間・余裕日数計算は未実装。

## 5. リスク管理
### 【実装済・完了】
- **リスク管理CRUD**: 名称、説明、影響度（大/中/小）、発生確率（高/中/低）、状態（未対応/対策中/監視中/発生/クローズ）の追加・編集・削除機能。
- **優先度の自動判定**: 影響度×発生確率から `critical` / `high` / `medium` / `low` を自動判定するロジック（`calculate_priority`）のバックエンド実装。
- **リスク対応策（カウンターメジャー）管理**: 各リスクに紐づく対応策（対策内容、担当者、期限、ステータス）の一覧表示、新規作成、インラインでの編集、削除機能。

### 【未実装・相違箇所（課題）】
- **リスクヒートマップ表示**: ダッシュボード等での視覚的なリスク分布ヒートマップは未実装。
- **リスクから課題（Issue）への自動変換**: リスク状態を「発生」に変更した際に自動的に課題を作成するトリガーロジックは未実装。
- **スケジュールバッファ自動提案**: リスクの大きさに基づく日程バッファの自動追加・提案機能は未実装。

## 6. 課題（Issue）管理
### 【実装済・完了】
- **課題CRUD**: 課題の名称、説明、種別（バグ/仕様変更/要望/障害/その他）、優先度（緊急/高/中/低）、状態（未対応/対応中/解決済み/却下/クローズ）、担当者、期限の追加・編集・削除機能。

### 【未実装・相違箇所（課題）】
- **カンバンボード表示（Level 5）**: 状態別のカンバンボードは未実装。
- **タスク・成果物・リスクとの関連付け**: 課題がどのタスクに影響を及ぼしているか、どの成果物・リスクと関係しているかを選択・表示するマッピング機能は未実装。

## 7. 成果物管理
### 【実装済・完了】
- **成果物CRUD**: 各タスクに紐づく成果物（名称、種別、ファイルパス/URL、バージョン、状態）の追加・編集・削除機能。
- **成果物の状態自動連携**: 成果物の作成・登録時のステータスをデフォルトで「レビュー待ち（`pending`）」で生成し、即時レビュー可能な状態へする機能。
- **レビュー承認フロー**: 評価者（PM等）による成果物のレビュー（承認/却下）とレビューコメントの保存機能。
- **評価者情報の連動表示**: レビューを行った評価者（Reviewer）のユーザー名を成果物管理画面、およびタスク詳細画面の両方で連動して表示。
- **タスク画面からの追加・参照**: タスク詳細パネルから、該当タスクに紐づく成果物の一覧表示、およびそのタスクを親とした新規成果物追加機能。

### 【未実装・相違箇所（課題）】
- **「全成果物完了によるタスク完了」のハードチェック制限**:
  - 設計書の「全成果物が完了状態であることをタスク完了条件にする」制限は、システム的にバリデーション強制されておらず、手動で完了遷移させることが可能です（運用上カバーする状態）。

## 10. プロトタイプデモ画面（demo/index.html）との視覚的・機能的な相違点

`demo/index.html` に組み込まれている静的プロトタイプ用の表示仕様・ビジュアル要素のうち、現状の React 開発環境で未実装あるいは簡略化されている相違点は以下の通りです。

### 10.1. タスク管理画面での「ガントチャート（Gantt）」の描画
- **デモ画面 (`demo/index.html` の tasks)**:
  - 画面左側にタスクのツリー、右側にガントチャートのタイムラインバーが横並びで表示され、クリティカルパス上のタスクが赤枠で強調されるようにモックアップが組まれています。
- **現状の React 実装**:
  - 現在の `TasksPage.tsx` では、タスクを親子・孫の3階層インデントツリー形式でリスト表示するグリッド構造のみになっており、右側のタイムライン（ガントバー）のビジュアル描画は実装されていません。

### 10.2. 課題管理での「カンバンボード」の表示
- **デモ画面 (`demo/index.html` の issues)**:
  - 「未対応」「対応中」「解決済」「クローズ」の4カラムから構成される視覚的なカード型のカンバンボード（`class="board"`）が配置されています。
- **現状の React 実装**:
  - 現在の `IssuesPage.tsx` では、課題一覧をテーブルリスト形式で表示し、編集や削除ボタンで操作する形に留まっており、ドラッグ＆ドロップなどのカンバンボード形式のUIは実装されていません。

### 10.3. タスク対話型「状態フロー図（State Flow）」モーダル
- **デモ画面 (`demo/index.html` の stateFlowModal)**:
  - 選択したタスクの現在の状態（未着手・対応中など）を起点とし、次に遷移可能な矢印を動的に点滅表示させる SVG 形式のフロー図インタラクションが実装されています。
- **現状の React 実装**:
  - 未実装です。タスク詳細での状態遷移はドロップダウンの選択によってのみ制御されています。

### 10.4. プロジェクト全体対話型「全体状態フロー図（Project Flow）」モーダル
- **デモ画面 (`demo/index.html` の projectFlowModal)**:
  - プロジェクトのタスク依存関係や進行状態を、カラーノード（状態別）および赤枠＋星マーク（クリティカルパス）で接続した、全体フローのインタラクティブな SVG トポロジー図を表示するモーダルが配置されています。子タスクノードをクリックすると、その階層の下位タスク一覧へと再帰的にドリルダウン・ロールバックできる仕組みがあります。
- **現状の React 実装**:
  - 未実装です。クリティカルパスの自動算出および視覚的ツリー図の描画はまだ行われていません。

### 10.5. ダッシュボードの「リスクヒートマップ（Heatmap）」
- **デモ画面 (`demo/index.html` の dashboard)**:
  - 影響度（大中小）×発生確率（高中低）の3×3のグリッド形式で、各マトリクスに存在するリスク件数を赤・オレンジ・黄色のカラーグラデーションで分布表示する「リスクヒートマップ」が配置されています。
- **現状の React 実装**:
  - 現在の `ProjectDashboardPage.tsx` では、プロジェクトのリスク全体の件数（大中小など）の数値集計は行われていますが、3×3マトリクスのカラーグリッドヒートマップの描画は実装されていません。
